# UIUX監査レポート — 2026-07-02

調査手法: ①コードベース全読（src/app・components・lib）②本番サイトPlaywright実操作 ③地図APIコスト調査（Web検証済み）④競合UX分析（サウナイキタイ・KUDOCHI・食べログ等）

---

## TL;DR

1. **地図は既にGoogle Mapsではない**。Leaflet + OpenStreetMap（`tile.openstreetmap.jp`）で実装済みで、APIキー不要・費用ゼロ。「Google Mapsは費用がかかるから使えない」は誤解で、仮にGoogle Mapsを使っても月10,000マップロードまで無料（現在の規模なら余裕で無料枠内）。**費用は問題ではなく、品質（クラスタリング未実装によるマーカー重なり）が問題**。
2. **最大のUX破綻は検索ページの地図**。全国表示で価格ラベルが数百個重なり判読不能な塊になっている。/search はGA4で最多PVのページであり、第一印象を最も損ねている箇所。
3. **「なぜ使われないか」の本質は、一回使ったユーザーが戻ってくる理由がないこと**。検索→公式サイトへ離脱で完結し、空き状況なし・口コミなし・お気に入りは端末ローカルのみ。個室サウナは「予約して行く」ものなのに、予約の意思決定をこのサイトで完結できない。
4. **勝ち筋は「個室サウナのスペック比較・予約可否の即答性」**。競合調査の結果、個室サウナ特化で予約情報を統合したプラットフォームは存在しない（空白地帯）。サウナイキタイと口コミで戦わず、「2人なら1人あたり¥X」「今日空いてる」に全振りする。

## 1. 現状評価 — できていること（意外と多い）

コード・実操作の両面で確認した強み。壊さないこと。

- トップの4ステップ検索フォーム（誰と/どこ/いつ/いくら）は初見でも迷わない。駅名オートコンプリート・現在地検索・ARIA対応まで実装済み
- 施設詳細ページの回遊導線（近くの施設・似た価格帯・条件リンク・関連記事）は非常に作り込まれている
- **1人あたり料金の算出表示**（詳細ページ）は競合にない差別化要素の芽
- モバイルのフィルタボトムシート・全画面地図オーバーレイ・固定予約CTAバーなど、モバイル固有UIは水準以上
- 予約導線は `/go/{id}` リダイレクト+UTM付与で計測可能な設計
- 表示速度は体感良好、コンソールエラーもほぼゼロ（後述の2件を除く）
- デザインは「素人っぽさ」を感じさせない。カード設計・情報密度・余白は良好

## 2. なぜ使われないか（仮説と根拠）

### 2-1. 一回きりで終わる構造（リテンションゼロ設計）
- ユーザーは検索→施設決定→公式サイトへ離脱で用が済む。**再訪する理由が設計されていない**
- お気に入りはlocalStorageのみ。機種変・別端末で消える。共有もできない
- ターゲット（カップルのデート利用）を考えると致命的: デートの店選びは**2人で共有して決める**行為なのに、お気に入りリストを相手に送れない

### 2-2. 意思決定を完結できない
- 個室サウナは予約必須の業態。なのに**空き状況が一切分からない**（timeSlotsは3施設のみ、しかも静的な「目安」）
- サイト内の随所に「※最新の料金は公式サイトでご確認ください」→ 結局公式サイトを見る必要がある＝**サイトの存在価値を自ら打ち消している**
- 競合調査の結論: 検索プラットフォームで予約情報を統合した例は個室サウナ領域に存在しない。ここが空白

### 2-3. 信頼の担保がない
- 口コミ・評価ゼロ。代替のサウナ子コメント・施設紹介文に自動生成の痕跡（「カップルでの利用もOKし、グループ利用にもぴったりし」等の不自然な接続）が残っており、**人間が確かめた情報という実感がない**。E-E-A-T的にもマイナス
- 情報の鮮度表示（この施設情報はいつ確認したか）がない

### 2-4. 検索ページ（最多PVページ）の視覚的破綻
- 全国表示で価格ラベルマーカーが関東圏で数百個重なり、黒い塊になる（`.playwright-mcp/02c-search-check.png`）
- リスト（安い順・東京の施設）と地図（全国〜甲信越表示）の初期状態がズレていて「地図が使い物にならない」印象
- 件数表示の二重化: 「この範囲に193件（全478件中）」と「478件の個室サウナが見つかりました」が同時に見え混乱する
- React hydration error #418 が/searchで発生（表示崩れの潜在リスク）

### 2-5. サウナイキタイで足りてしまう
- 個室サウナもサウナイキタイに載っている。月間240万PV・口コミ資産で正面勝負は不可能
- 「個室特化だから何が良いのか」（貸切・カップルOK・料金の即答性）がUI上で際立っていない

## 3. 地図機能の結論（費用の質問への回答）

### 事実
| 項目 | 内容 |
|------|------|
| 現在の実装 | Leaflet + react-leaflet + OSMタイル（`FacilityMap.tsx`）。**費用ゼロ・APIキー不要** |
| CLAUDE.mdの記載 | `@react-google-maps/api` 使用と書いてあるが**実コードでは未使用**（記載が古い） |
| Google Maps現行料金（2026年） | Essentials階層で**月10,000マップロード無料**、超過$7/1,000。月1,000PVなら完全無料 |
| Google Maps Embed API | 完全無料・無制限 |
| 現タイルサーバーのリスク | `tile.openstreetmap.jp` はOSM日本コミュニティのボランティア運営。SLAなし、商用大量利用の保証なし |

### 推奨
- **費用を理由に地図を諦める必要は一切ない**。月10万PVに成長しても、MapLibre+OpenFreeMap（完全無料）、MapTiler Flex（$25/月）、Geolonia Pro（¥4,378/月・日本品質最高）等の選択肢がある
- 本当に直すべきは**クラスタリング未実装**と**初期ビュー**（→ P0提案参照）
- 中期では **MapLibre GL JS + OpenFreeMap（ベクタタイル）** への移行を推奨: 無料のまま、地図の見た目をブランドカラーに合わせてカスタマイズでき、「安っぽいOSM感」を消せる。タイルサーバーの安定性も上がる

## 5. 破壊的変更を許容する場合の最適案

**「個室サウナ検索サイト」から「個室サウナの比較・予約可否即答ツール」への転換。**

- 情報設計の軸を「エリア→施設一覧」から「**シチュエーション→比較→予約**」へ:
  - トップの主語を「誰と行く？」からさらに進め、「2人で今週末、予算1人3,000円」で**比較表が即出る**体験に
  - 検索結果のデフォルトを「1人あたり料金順」に（カップル・グループ前提の並び）
- 検索ページをアプリのメイン画面として再設計: モバイルはリスト⇄地図をタブで切替（現状のフローティングボタンより発見性が高い）+ ボトムナビ
- 口コミは作らない（サウナイキタイに勝てない・コールドスタート問題）。代わりに**スペックの正確性・鮮度・比較性**で信頼を取る
- 空き状況はCoubic対応施設から段階導入し、「空きが分かる個室サウナ検索」を将来の一枚看板に育てる

**理由**: 競合調査で確認した市場構造 — 口コミ×網羅はサウナイキタイが独占、施設直営の予約サイトは自社店舗のみ、**「横断比較×予約可否」は誰もやっていない**。saunakoの既存資産（487施設の構造化データ、料金プラン、1人あたり料金算出、/go計測基盤）はすべてこの方向に活きる。
